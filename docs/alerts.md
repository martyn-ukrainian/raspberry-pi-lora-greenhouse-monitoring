# Алерти — Фаза 1

## Що це

Логіка виявлення "щось не так у теплиці" і надсилання оператору повідомлення у Telegram простою українською. У Фазі 1 система тільки **сповіщає** — жодних автоматичних дій (полив, вентилятори). Оператор отримує повідомлення й діє руками.

## Пороги — за теплицею, не за культурою

Кожна теплиця має **власний набір порогів**. Причини:

- **Матч ментальної моделі оператора.** Оператор думає "у теплиці №2 щось не так", а не "профіль огірків має відхилення".
- **Реальність нерівномірна.** Дві теплиці з огірками можуть мати різні пороги: нова двоплівкова тримає тепло краще за стару, стара має протяги, тощо.
- **Без зайвого шару непрямоти.** Просто "вимір → знайти пороги теплиці".

Стартові значення беруться з [`журнал.md`](./журнал.md) і потім **коригуються за сезон на фактичних вимірах** окремо для кожної теплиці. Фізично конфіг лежить у `server/config/thresholds.yaml`.

## Дві незалежні механіки анти-спаму

Без обмежень система би слала повідомлення на кожен вхідний вимір, поки стан не повернеться в норму. Це неприйнятно. Тому дві окремі фільтрації:

### A. Витримка перед алертом (`dwell_minutes`)

Перш ніж згенерувати алерт, стан "поза межами" має **триматись протягом X хвилин** підряд. Захищає від разових стрибків (провітрили двері на 30 секунд — температура впала — знову норма).

**Різні датчики — різні витримки:**
- Температура повітря стрибає швидко → коротша витримка (3-5 хв).
- Вологість повітря інерційніша → середня (5-10 хв).
- Вологість ґрунту дуже інерційна → довша (15-30 хв). Якщо ґрунт справді сухий, це не за секунди.

### B. Cooldown після алерта (`cooldown_minutes`)

Якщо алерт уже пішов — не повторюй ще Y хвилин, навіть якщо стан не змінився. Захищає від "я вже сказав, оператор побачив".

**За замовчуванням 30 хвилин**, глобально для всієї системи. За потребою можна переоприділити per-sensor.

### Як вони працюють разом

```
Час →   0    3    5    10   15   20   30   40   50   60   70
Стан:   ok   ok   BAD  BAD  BAD  BAD  BAD  BAD  BAD  BAD  BAD
                  ↑
                  почалось погано
                            ↑
                            минуло 5 хв ≥ dwell_minutes → ПЕРШИЙ АЛЕРТ
                                                  ↑
                                                  cooldown 30 хв — тиша
                                                                        ↑
                                                                        cooldown вийшов
                                                                        якщо все ще BAD → новий алерт
```

## Форма конфіга (приклад)

```yaml
# Глобальні дефолти для витримки і cooldown.
# Можна перекрити per-sensor у блоках теплиць нижче.
defaults:
  dwell_minutes: 5
  cooldown_minutes: 30

greenhouses:
  greenhouse-1:
    label: "Розсадна (огірки)"
    air_temperature:
      min: 24.0
      max: 28.0
      dwell_minutes: 3         # температура стрибає швидко, коротша витримка
    air_humidity:
      min: 75.0
      max: 90.0
      dwell_minutes: 10
    soil_moisture:
      min: 30.0
      dwell_minutes: 20        # ґрунт інерційний, довша витримка

  greenhouse-2:
    label: "Огірки основна №1"
    air_temperature: { min: 24.0, max: 28.0 }   # використовує defaults.dwell_minutes
    air_humidity: { min: 75.0, max: 90.0 }
    soil_moisture: { min: 30.0, dwell_minutes: 20 }

  greenhouse-3:
    label: "Помідори"
    air_temperature: { min: 20.0, max: 30.0 }
    air_humidity: { min: 60.0, max: 70.0, dwell_minutes: 15 }
    soil_moisture: { min: 25.0, dwell_minutes: 20 }
```

**Правило:** якщо поле `dwell_minutes` є на рівні датчика — воно перекриває `defaults.dwell_minutes`. Якщо нема — беруться дефолти. Так само з cooldown.

**Поле `label`** — людське імʼя теплиці. Використовується в повідомленнях замість технічного `greenhouse-2`.

## Стан у памʼяті сервера

Для роботи двох механік треба тримати мінімальний state в памʼяті:

```
state = {
    (node_id, sensor): {
        "out_since": datetime | None,       # коли вперше вийшло поза межі
        "last_alert_at": datetime | None,   # коли останній раз відправляли алерт
    }
}
```

**Логіка на кожен вхідний вимір:**

1. Береш пороги для `(node_id, sensor)`.
2. Якщо значення **в межах** → `out_since = None`, `last_alert_at` **не чіпаємо**.
3. Якщо значення **поза межами**:
   - Якщо `out_since is None` → зафіксувати `out_since = now`.
   - Інакше — обчислити скільки триває (`now - out_since`).
   - Якщо триває **менше** `dwell_minutes` → тиша, чекаємо.
   - Якщо триває **більше або рівно** `dwell_minutes`:
     - Якщо `last_alert_at` **None** або з нього минуло `≥ cooldown_minutes` → **шлемо алерт**, `last_alert_at = now`.
     - Інакше — тиша (cooldown ще діє).

При перезапуску сервера state забувається — це прийнятно для Фази 1. Максимум один "зайвий" алерт після рестарту.

## Коли перевіряти

На **кожен вхідний POST /measurements**. Логіка синхронна з отриманням даних, без окремих періодичних завдань. У масштабі 8 вузлів з кількома вимірами на годину — це нічого не тисне.

## Формат повідомлення

Отримувач — оператор без технічної підготовки, тому:

- **Без англійських термінів** (не "air_temperature", не "greenhouse-1").
- **Використовуємо `label` теплиці** з конфіга.
- **Прості порівняння з нормою**: не тільки "жарко", а "жарко: 36°, звичайно має бути до 28°".
- **Емодзі як візуальний якір** для типу проблеми.
- **Дія-підказка** якщо очевидна.
- **Можна згадати тривалість** аномалії: "вже 12 хвилин жарко".

Приклади:

- 🔥 *Теплиця "Огірки основна №1" — вже 8 хвилин жарко: 36°, звичайно має бути до 28°. Провітрити?*
- 🌡️ *Теплиця "Розсадна" — холодно: 21°, для огірків має бути мінімум 24°.*
- 💧 *Теплиця "Помідори" — сухо в ґрунті (23%) вже пів години. Треба полити.*
- 🍅 *Теплиця "Помідори" — вологість повітря 78% (норма до 70%). Ризик фітофтори — провітрити.*

Точні шаблони згенеруємо при реалізації.

## Доставка

- **Telegram Bot API**, один бот (створюється через @BotFather у Telegram).
- **Один отримувач** — `chat_id` оператора. Кілька отримувачів у Фазі 1 не робимо.
- **Токен бота і chat_id** живуть у `server/.env` — це секрети, не в `thresholds.yaml`.
- **Без ретраїв.** Якщо Telegram недоступний — логуємо помилку і йдемо далі.

|---|---|---|
| Пороги за теплицями | `server/config/thresholds.yaml` | Розробник (потім UI) |

## Що НЕ в цій фазі

- **Автоматичні дії.** Полив, вентиляція, актуатори — Фази 2 і 3.
- **Дашборд у веб.** Оператор не сяде за компʼютер.
- **Повідомлення "стало краще".** Тільки алерти "погано".
- **Групові чати / кілька отримувачів.** Один одержувач.
- **SMS-фолбек.** Якщо Telegram відпав — пропускаємо.
- **Історичні звіти.** Тільки поточні алерти.
- **Розумні алерти** (ML, тренди). Тільки статичні пороги + витримка.
- **Персистентний state.** State в памʼяті, після рестарту забувається.
- **UI для редагування конфіга.** Правити руками YAML.
- **Watchdog "давно не було виміру".** Якщо вузол замовк — не сповіщатимемо. Логіка додасться коли зʼявиться реальний досвід збоїв LoRa.
- **Шаблони порогів за культурою (YAML-якорі або двошарова структура).** Зараз кожна теплиця повторює свій блок. Коли теплиць стане 6+ і дублювання стане болючим — впровадимо `crop_defaults` з якорями `<<: *cucumber`. YAML це підтримує з коробки.

Ці речі можна додати пізніше на основі реального досвіду.

## Майбутня еволюція (пост-Фаза 1)

Тут фіксуємо архітектурні напрямки, які **не робимо зараз**, але тримаємо в голові щоб рішення Фази 1 не заганяли нас у кут. Реалізується коли зʼявиться реальна потреба.

### Багатокористувацька платформа (мультитенант)

Замість "один оператор, `chat_id` в `.env`" — повноцінна SaaS-платформа. Юзери реєструються через веб, кожен бачить тільки свої теплиці. Ключові моделі:

- **`users`** — email, password_hash, created_at.
- **`telegram_links`** — `user_id → chat_id`, отримується через **deep linking**.
- **`greenhouses`** — переносяться з YAML у БД, отримують `owner_user_id`.
- **`thresholds`** — теж у БД, повʼязані з `greenhouse_id`.
- **`measurements`** — додається `greenhouse_id` як FK для ізоляції.

**Механіка зв'язування Telegram (deep linking):**

1. Юзер тисне "Підключити Telegram" на веб-платформі.
2. Сервер генерує тимчасовий токен `linktoken_abc123xyz` (дійсний 10 хв), запамʼятовує `user_id`.
3. UI відкриває `https://t.me/YOUR_BOT?start=linktoken_abc123xyz`.
4. Telegram відкриває бота і шле `/start linktoken_abc123xyz`.
5. Бот отримує токен + `chat_id`, шле на сервер → сервер звʼязує `user_id ↔ chat_id`.
6. Бот відповідає "✅ Готово", у платформі юзер бачить "підключено".

Стандартний паттерн, добре гуглиться як "**Telegram bot deep link start parameter**".

**Наслідок для UI:** веб-платформа — це **тонкий шар керування** над API. Уся логіка (детекція, роутинг сповіщень, зберігання, авторизація) — на сервері. UI просто робить HTTP-запити й показує стан. Тому потім можна мати різні клієнти (веб, мобільний, CLI) — усі до того самого API.

### Декілька каналів сповіщень (Notifier-паттерн)

Замість жорсткого "тільки Telegram" — абстрактний інтерфейс з різними реалізаціями. Паттерн Strategy/Adapter.

**Структура:**

```
alerts.py                    ← детекція (як зараз)
notifiers/
  __init__.py
  base.py                    ← абстрактний Notifier (протокол або ABC)
  telegram.py                ← реалізація для Telegram
  email.py                   ← реалізація для email (SMTP)
  push.py                    ← мобільний push (FCM / APNs)
  slack.py                   ← Slack webhook
  webhook.py                 ← довільний webhook (для інтеграцій)
```

**Ключова ідея:** `alerts.py` не знає *як* слати. Викликає `notifier.send(alert)` — де `notifier` вибирається з налаштувань юзера.

**Наслідок для моделі:**

```
user_notifiers
  user_id (FK)
  channel_type   -- 'telegram', 'email', 'push', ...
  config (JSON)  -- chat_id / email address / device_token / ...
  enabled
  min_severity   -- слати тільки алерти від рівня X (у майбутньому)
```

Юзер у UI: "Хочу Telegram + email на робочу пошту". Обидва зберігаються, сервер шле паралельно.

**Додавання нового каналу** = один файл у `notifiers/` + запис у UI. Основний код не чіпаємо. Це і є користь від абстракції.

### Дві deployment-моделі одного коду

Проект має підтримувати **обидві** моделі розгортання, вибір через конфіг:

**Модель А — Self-hosted (open source).** Агроном клонує репо, стартує на своєму Pi, реєструє свого Telegram-бота через @BotFather, кладе token в `.env`. Дані повністю у власника. Ідеально для приватності й безкоштовно.

**Модель Б — Central SaaS.** Одна велика платформа хостить багато користувачів, використовує **один Telegram-бот** (`@AgroMonitorBot` як приклад). Кожен користувач ізольований на рівні даних (row-level security).

**Ключове:** один і той самий код-база обслуговує обидві моделі. Конкретна модель задається конфігом. Це вимагає:

- **Notifier-абстракція з першого дня** (див. секцію вище). Endpoint кличе `notifier.send(alert)`, не `telegram.send(...)`.
- **Wiring через конфіг:** `NOTIFIER=telegram_direct` vs `NOTIFIER=telegram_hub` у `.env`.

### Bot Hub — окремий мікросервіс для "один бот, багато серверів"

Для Моделі Б і опційно для Моделі А (якщо агроном не хоче створювати свого бота через @BotFather).

**Паттерн:**

```
                ┌─────────────────────────┐
                │  agro-bot-hub (SaaS)    │
                │  Telegram Bot API       │
                │  Route: token → server  │
                └──────┬──────────────────┘
                       │
     ┌─────────────────┼─────────────────┐
     ▼                 ▼                 ▼
┌────────┐        ┌────────┐        ┌────────┐
│agro-#1 │        │agro-#2 │        │agro-#3 │
└────────┘        └────────┘        └────────┘
```

**Механіка:**

1. Локальний агро-сервер звертається до hub-у за link-токеном для юзера.
2. Hub повертає токен + deep link `t.me/AgroBot?start=hub_token_xyz`.
3. Юзер клікає, бот отримує `/start`, hub записує `(token → server_url, chat_id)`.
4. Коли локальний сервер хоче слати повідомлення юзеру → POST в hub API, hub передає в Telegram.

**Стек hub-у:** маленький FastAPI-сервіс (~200 рядків) + Redis для токенів + PostgreSQL для реєстру серверів і юзерів. **Окремий репозиторій**, окремий деплой.

**Реалізація в цьому проекті:** новий notifier `notifiers/telegram_hub.py`, який замість прямого виклику Telegram Bot API робить HTTP-запит на hub. Той самий інтерфейс `send(alert)`, інша реалізація.

### Що ще варто памʼятати

- **Ізоляція даних.** Кожен endpoint має перевіряти `owner_user_id == current_user.id`. Row-Level Security на рівні БД (Postgres підтримує з коробки) — правильний підхід, коли переростемо SQLite.
- **Історія алертів.** Зараз алерт — це "надіслати повідомлення". У платформі — це ще й **запис у БД** (`alerts` таблиця), щоб можна було переглядати "які алерти в мене були за тиждень".
- **Rate limiting per user.** Cooldown зараз глобальний. У мультитенант-системі — per-user, per-greenhouse, per-sensor.

## Roadmap реалізації

1. Створити `server/config/thresholds.yaml` з блоком для кожної теплиці.
2. Написати `server/thresholds.py` — модуль що завантажує YAML і валідує його через Pydantic. Ловить опечатки в назвах теплиць, пропущені поля, невалідні значення (максимум < мінімум).
3. Написати `server/alerts.py` — логіка стану `(node_id, sensor) → out_since / last_alert_at` + функція `check(measurement)` яка повертає алерт або None.
4. Створити Telegram-бота через @BotFather, отримати token, покласти в `.env`.
5. Написати `server/notifiers/base.py` — `Notifier`-протокол з методом `send(alert)`. Реалізації живуть у `server/notifiers/`. Для Фази 1 — тільки `notifiers/telegram.py` як конкретна реалізація (прямий бот через @BotFather, `token`+`chat_id` з `.env`).
6. Підключити все в endpoint `POST /measurements` — після збереження в базу викликати `alerts.check(...)` і, якщо є алерт, викликати `notifier.send(alert)` через сконфігурований в `main.py` `notifier`-обʼєкт.
7. Прогнати через симулятор — генерувати "аномальні" виміри навмисно і дивитись, чи приходять повідомлення з правильним таймінгом (не одразу, а після витримки).

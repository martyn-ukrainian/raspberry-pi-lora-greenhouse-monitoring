# Архітектура системи

## Загальна ідея

Система складається з трьох шарів, які фізично й технологічно відрізняються: вузли в теплицях, шлюз і сервер. Дані рухаються в одному напрямку — від датчиків до оператора в Telegram. Кожен шар можна розробляти й тестувати окремо.

## Шар перший — вузли в теплицях

У кожній теплиці стоїть автономний вузол на базі Heltec WiFi LoRa 32 V3. Це мікроконтролер ESP32-S3 із вбудованим LoRa-чипом SX1262. До нього під'єднані датчики: SHT31 вимірює температуру й вологість повітря, а ємнісний датчик вологості ґрунту версії 1.2 стежить за землею. Вузол періодично прокидається, знімає показники й передає їх по радіо, після чого знову засинає для економії енергії.

Прошивка пишеться на C++ (Arduino або PlatformIO). Оскільки вузол — це слабкий мікроконтролер без операційної системи, іншого варіанту, крім C чи MicroPython, тут нема. Прошивка єдина й параметризована для всіх вузлів, а не окрема під кожен тип теплиці — це спрощує підтримку.

Ближні вузли живляться від мережі кабелем, дальні — від сонячної панелі з акумулятором LiFePO4 (це вже друга фаза).

## Шар другий — шлюз

У розсадній теплиці, де є електрика, стоїть окремий вузол-шлюз. Він не має датчиків — його завдання приймати LoRa-повідомлення від усіх теплиць і передавати їх на сервер. Шлюз під'єднаний до Raspberry Pi через USB. Топологія проста — зіркова, з власною адресацією вузлів, без складнощів повноцінного LoRaWAN. На масштабі 6–14 вузлів цього цілком достатньо.

## Шар третій — сервер на Raspberry Pi

Raspberry Pi 5 — це повноцінний Linux-комп'ютер, а не мікроконтролер. Він працює цілодобово й тримає на собі весь бекенд. На ньому крутиться застосунок на Python (FastAPI), який приймає дані від шлюзу, зберігає їх у базу SQLite і віддає назад на запит. Поверх цього працює логіка порогів: коли показник виходить за межі норми, сервер формує зрозуміле повідомлення й надсилає його оператору через Telegram-бота.

Важливо, що код сервера однаковий на комп'ютері розробника й на малині — Python кросплатформний. Тому бекенд спершу пишеться й тестується на ПК за допомогою симулятора вузла, а коли Raspberry Pi готова, папка server просто копіюється на неї й запускається без змін.

## Потік даних

Датчик знімає показник. Вузол передає його по LoRa на шлюз. Шлюз віддає дані через USB на Raspberry Pi. FastAPI приймає запит, валідує його й записує в SQLite. Логіка порогів перевіряє значення й, якщо потрібно, шле оператору Telegram-повідомлення. Усі виміри зберігаються з першого дня — це фундамент для майбутніх емпіричних порогів і графіків.

## Принципи

Спершу довести весь ланцюг на одному вузлі, і лише потім масштабувати — не нарощувати кількість передчасно. Починати з софту, поки залізо їде: бекенд можна писати й перевіряти ще до приїзду вузлів. Логувати всі дані з самого початку. Автоматизацію кватирок (зараз це ручний механізм на мотузці) поки відкладено як надто дорогу — замість неї Telegram підказуватиме оператору діяти вручну.

## Розподіл технологій

Вузол у теплиці працює на Heltec V3 і програмується на C++. Сервер і хаб — це Raspberry Pi 5 на Python з FastAPI. Це два різні шари: C++ відповідає за залізо й радіо, Python — за зберігання, логіку й сповіщення.
